# データ保存の安全性

## 直していた問題

### 1. 書き込みが非アトミックだった

`File.WriteAllText` は既存ファイルを**切り詰めてから**書く。
書き込みの途中で強制終了・電源断・スリープ復帰の失敗が起きると、
`schedules.json` が中途半端な内容になるか、0バイトになる。

保存はアイテムを1つ変えるたびに全件書き込みで、
ドラッグ確定・記録停止・編集のたびに走っていたため、窓は常時開いていた。

### 2. 壊れたデータがサンプルデータで上書きされていた（最も危険）

```csharp
// 修正前
public static ObservableCollection<ScheduleItem> LoadData()
    => JsonFileRepository.LoadFromFileSync<...>(ScheduleFilePath) ?? LoadSampleData();
```

`LoadFromFileSync` は例外を握りつぶして `null` を返していた。
呼び出し側はそれを「ファイルが無い＝初回起動」と解釈し、**サンプルデータ**を読み込む。

つまり JSON が1バイトでも壊れると:

1. 起動時に実データがサンプル（朝会・週次レビュー・顧客訪問…）に置き換わる
2. ユーザーが何か操作した瞬間、そのサンプルが `schedules.json` に**保存される**
3. 実データは完全に失われ、しかもエラーは一切表示されない

## 対策

### 書き込みのアトミック化

```
一時ファイルへ完全に書き出す  →  File.Replace で本体と入れ替える
                                 （旧内容は .bak へ退避）
```

一時ファイルへの書き込みが途中で死んでも本体は無傷。
`File.Replace` が使えない環境では「退避 → 削除 → 移動」の順にフォールバックする
（順序が重要で、必ず退避を先に済ませる）。

### 3段のバックアップ

| ファイル | 内容 | 救えるケース |
|---|---|---|
| `schedules.json` | 本体 | — |
| `schedules.json.bak` | 直前の保存内容 | 書き込み中の破損 |
| `schedules.json.<日付>.bak` | その日の最初の保存前の内容（7世代） | 「昨日うっかり全部消した」 |

日次スナップショットは、直前世代では救えない
「気づかないまま何度も保存してしまった」ケースのために置いている。

### 読み込みの多段フォールバック

本体 → `.bak` → 日次スナップショット（新しい順）の順に試し、
結果を `LoadStatus` で呼び出し側へ返す。

| Status | 意味 | 挙動 |
|---|---|---|
| `NotFound` | ファイルが無い（真の初回起動） | サンプルデータを読み込む |
| `Loaded` | 正常 | そのまま |
| `RecoveredFromBackup` | 本体が壊れていたが復元できた | 警告バナーを出す。保存は継続 |
| `Failed` | 本体もバックアップも読めない | 警告バナーを出し、**保存を停止** |

`Failed` で保存を止めるのが要点。
ここで保存を許すと、壊れたファイルの上に「記録が無い」という結果を
確定させてしまい、手動復旧の余地まで消える。

空ファイルは「壊れている」扱いにしている。
中断された書き込みの結果である可能性が高く、正常な空データと区別できないため。

### 保存のデバウンス

変更のたびの全件書き込みを 700ms でまとめる。書き込み頻度が下がり、破損の窓も狭まる。

確定させるタイミング:

- タイマー満了（700ms）
- ウィンドウを閉じたとき（トレイへ隠すときも含む）
- アプリ終了時

クラッシュ時は最大700msぶんの変更が失われうるが、
従来の「破損して全部失う」に比べれば桁違いに小さい。

### ユーザーへの通知

異常時はツールバー下に警告バナーを出す。

- 何が起きたか（復元した／読めなかった）
- どのバックアップから復元したか
- 保存を停止していること（`Failed` のとき）
- 「データフォルダを開く」ボタン

従来は `Debug.WriteLine` のみで、ユーザーには何も見えなかった。

## 変更したファイル

| ファイル | 変更 |
|---|---|
| `Services/JsonFileRepository.cs` | アトミック書き込み、バックアップ、多段フォールバック、`LoadResult<T>` |
| `Services/FilePersistenceService.cs` | `LoadData` が状態を返すように。サンプル差し替えは `NotFound` のときだけ |
| `Services/SettingsService.cs` | 新しい戻り値への追随 |
| `ViewModels/MainViewModel.Data.cs` | デバウンス、`FlushDataSave`、読み込み失敗時の保存停止、通知 |
| `Views/MainWindow.xaml` | 警告バナー |
| `Views/MainWindow.xaml.cs` | 終了時・トレイ格納時のフラッシュ |

## 残る課題

- 設定ファイル（`settings.json`）とメモ（`memos.json`）は同じアトミック書き込みの
  恩恵を受けるが、読み込み失敗時の通知は予定データのみ
- バックアップからの復元をUIから選べない（現状は自動フォールバックと手動コピーのみ）
- 手動バックアップ／エクスポートの導線がない
